El primer diagrama muestra el tráfico de señalización SIP involucrado en el establecimiento, modificación y terminación de sesiones siguiendo el SIP RFC 3261 [https://tools.ietf.org/html/rfc3261] y cualquier RFC relacionado [https://www.packetizer.com/ipmc/sip/standards.html].

Estas son las entidades SIP externas involucradas:

  • UACs: teléfonos de hardware de usuarios, softphones, dispositivos compatibles con SIP.

  • Operadores SIP/Proveedores DDI: operadores utilizados para interconectar IvozProvider con redes SIP externas (y, probablemente, con la PSTN).

Todo el tráfico SIP (en cualquiera de los transportes soportados: TCP, UDP, TLS, WSS) que envían/reciben es hacia/desde estas dos entidades SIP internas de IvozProvider:

  • Proxy SIP de Usuarios (ejecutando Kamailio [https://www.kamailio.org]).

  • Proxy SIP de Trunks (ejecutando Kamailio [https://www.kamailio.org]).

De hecho, los UACs de usuarios solo hablan con el Proxy SIP de Usuarios y los ‘Operadores SIP’ y los ‘Proveedores DDI’.

Los proveedores solo hablan con el Proxy SIP de Trunks.

Dentro de IvozProvider, estos dos proxies pueden hablar con Servidores de Aplicaciones ejecutando Asterisk [http://www.asterisk.org/] para algunos tipos de clientes, pero no se permite que ningún elemento externo hable directamente con los Servidores de Aplicaciones.

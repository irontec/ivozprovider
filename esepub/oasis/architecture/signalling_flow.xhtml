El primer diagrama muestra el tráfico de señalización SIP que ocurre en el establecimiento, modificación y finalización de sesiones acorde al RFC 3261 [https://tools.ietf.org/html/rfc3261] y otros RFCs relacionados [https://www.packetizer.com/ipmc/sip/standards.html].

Estas son las entidades SIP externas involucradas:

  • UACs: teléfonos de usuarios, softphones, dispositivos compatibles con SIP.

  • Proveedores SIP: proveedores usados para la interconexión de IvozProvider con redes externas SIP (y, probablemente, con la PSTN).

Todo el tráfico SIP (en cualquiera de los transportes soportados: TCP, UDP, TLS, WSS) que se envia/recibe desde la plataforma se hará deste las siguiente dos entidades SIP internas de IvozProvider:

  • Proxy SIP de Usuarios (ejecutando Kamailio [https://www.kamailio.org]).

  • Proxy SIP Proveedores (running Kamailio [https://www.kamailio.org]).

De hecho, los UACs de usuario solo se comunican con el Proxy SIP de usuarios y los ‘Proveedores SIP’ solo se comunican con el Proxy SIP de Proveedores.

Dentro de IvozProvider estos dos proxies hablan a los Servidores de Aplicación que ejecutan Asterisk [http://www.asterisk.org/], ya que ningun elemento externo puede hablar con los Servidores de Aplicación directamente.
